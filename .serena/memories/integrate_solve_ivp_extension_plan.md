# Notes for adding `solve_ivp_osc` using C++ backend (riccaticpp)
- Current IVP API lives in `scipy/integrate/_ivp/ivp.py`: `solve_ivp` dispatches via `METHODS` dict mapping method names to solver classes (`RK23`, `RK45`, `DOP853`, `Radau`, `BDF`, `LSODA`). These solver classes are pure Python wrappers around Fortran/C solvers in `_ivp/*.py` or legacy `_ode`/`_odepack` modules.
- Legacy compiled solvers are exposed through `_ode` (imports compiled `_vode`, `_dop`, `_odepack`) and `_odepack_py.odeint` is a wrapper around `_odepack` C ext. All compiled modules are imported in `scipy/integrate/__init__.py`, so new APIs must be added there to be public.
- To add a C++ solver:
  1) Create a Python extension module (likely `_riccati_cpp` or similar) built via `py3.extension_module` in `scipy/integrate/meson.build`, with sources moved under `scipy/integrate/` (currently in `scipy/integrate/riccaticpp`). Include `dependencies: [np_dep, ccallback_dep]` (and BLAS/LAPACK deps if needed). Add `link_args: version_link_args` and rely on project-wide `cpp_std=c++17`. Use `.cpp` sources or set `cpp_args`/`link_language: 'cpp'` if necessary.
  2) Provide a thin Python wrapper (pure Python or Cython) that imports the extension and exposes a clean call signature. Place it under `scipy/integrate/_ivp/` or top-level integrate module, and install via `py3.install_sources` if pure Python.
  3) Export the new API from `scipy/integrate/__init__.py` and update `__all__`/autosummary docs. If integrating with the existing solver registry, add an entry or create a dedicated function `solve_ivp_osc` that invokes the C++ backend directly.
- Build/test workflow: run `source .venv/bin/activate && ~/.pixi/bin/pixi run build` to compile the extension and `~/.pixi/bin/pixi run test ./scipy/integrate/` for regression. Extensions are installed into `build-install/.../site-packages` during pixi test runs.
- Reuse `ccallback.h` for callbacks from C++ to Python (`LowLevelCallable`) if the backend needs user-supplied RHS functions, mirroring `_quadpack` and `_odepack` patterns.
